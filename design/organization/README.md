# 组织管理模块

## 模块概述

组织管理模块负责企业组织架构和员工管理，包括组织架构树（公司、分支机构、部门、班组）、岗位定义、技能定义，以及**人类员工**的管理。

**重要概念**：
- **人类员工**：真实的人，存储在 `t_employees` 表，在**员工管理**子模块中管理
- **数字员工**：AI Agent，存储在 `t_agent` 表，在 **Agent 管理模块**中管理
- 两者是**平行关系**，共同构成企业员工体系

## 核心职责

| 职责 | 说明 |
|------|------|
| **组织管理** | 企业组织架构维护（公司、分支机构、部门、班组） |
| **岗位管理** | 人类员工的岗位定义（职责、数据权限） |
| **技能定义** | 数字员工的技能组合定义（org-skill） |
| **员工管理** | 人类员工的统一管理 |
| **Agent 绑定** | 为人类员工分配专属或共享 Agent |

## 数据模型关系

### 整体架构

```
组织管理
├── 组织 (t_departments)
│   ├── 公司（根节点）
│   ├── 分支机构
│   ├── 部门
│   └── 班组
│
├── 岗位 (t_positions)
│   └── 可定义在任意组织层级
│       ├── 岗位职责（profile_json）
│       └── 数据权限范围（data_scope）
│
├── 技能定义 (t_departments_skill)
│   └── 可定义在任意组织层级
│       ├── 职责范围（responsibilities）
│       └── 可用 Skill 列表（skills）
│
├── 员工 (t_employees) - 人类员工
│   ├── 关联组织（org_id）
│   ├── 关联岗位（position_id）
│   ├── 关联 t_users（系统管理）
│   └── 可绑定 Agent（专属/共享）
│
└── Agent (t_agent) - 数字员工
    ├── 关联组织（org_id）
    ├── 关联技能定义（skill_id）
    ├── 服务模式（专属/共享/公共）
    └── 可绑定用户（专属模式）
```

### 人类员工与数字员工的关系

```
┌─────────────────────────────────────────────────────────┐
│                    企业员工体系                          │
├─────────────────────────────────────────────────────────┤
│  人类员工（t_employees）        数字员工（t_agent）         │
│       │                              │                  │
│       │ 专属模式：bind_user_id       │                  │
│       └──────────────────────────────┘                  │
│              一对一绑定                                  │
│                                                          │
│       │                              │                  │
│       │ 共享/公共模式：多对多使用    │                  │
│                                                          │
├─────────────────────────────────────────────────────────┤
│  共同点：                                                │
│  - 都归属组织节点（org_id）                              │
│  - 都基于组织树管理                                      │
│  - 成本都计入所属部门                                    │
├─────────────────────────────────────────────────────────┤
│  区别：                                                  │
│  - 人类员工：随入职/转岗/离职变动                        │
│  - 数字员工：沉淀岗位能力，不随人员变动                  │
└─────────────────────────────────────────────────────────┘
```

## 组织架构设计

### 组织类型

| 类型 | 说明 | 层级 | 示例 |
|------|------|------|------|
| company | 公司/总部 | 0（根节点） | 示例科技有限公司 |
| branch | 分支机构 | 1 | 北京分公司、上海办事处 |
| departments | 部门 | 2 | 技术部、人事部 |
| group | 班组/小组 | 3+ | 后端开发组、销售一组 |

### 组织架构示例

```
示例科技有限公司（company）
├── 北京分公司（branch）
│   ├── 技术部（departments）
│   │   ├── 后端开发组（group）
│   │   ├── 前端开发组（group）
│   │   └── 测试组（group）
│   ├── 人事部（departments）
│   └── 财务部（departments）
├── 上海分公司（branch）
│   ├── 技术部（departments）
│   └── 销售部（departments）
└── 广州办事处（branch）
    └── 销售部（departments）
```

## 关键设计原则

### 1. 岗位能力沉淀

Agent 归属组织节点，沉淀该岗位的能力和经验：
- **不随人员变动而变动**：人员转岗时，Agent 留在原岗位
- **成本归属明确**：API/算力费用计入 Agent 所属部门
- **知识传承**：岗位经验通过 Agent 传承给接替者

### 2. 数据权限分离

- **Agent 决定能力范围**：通过 org-skill 定义可用 Skill
- **用户决定数据权限**：通过岗位（t_positions）的数据权限范围（data_scope）控制

### 3. 服务模式灵活

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **专属** | 一对一服务 | 高管个人助理 |
| **共享** | 多用户共享 | 部门协作助手 |
| **公共** | 全组织可用 | 公共服务（IT/HR支持） |
